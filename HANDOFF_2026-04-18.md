## 1. The root-cause bug that took the whole session to find

**`handler.py` uses `TestClient(app)` with default `follow_redirects=True`.** Every FastAPI handler returning `RedirectResponse` (OAuth start endpoints, PayPal return, billing portal, thumbnail shares, YouTube callback) gets its 302 silently followed by TestClient INSIDE the RunPod worker. TestClient resolves the Location target as an in-process path:
- `https://studio.nyptidindustries.com/?youtube=error&...` → path becomes `/` → matches `@app.get("/")` which serves Studio's SPA `index.html`.

So the browser that should receive `302 → accounts.google.com/o/oauth2/auth` instead receives **200 + Studio SPA HTML**. Every redirect-based flow dead-ends on the Studio landing page.

Fix (committed, not yet in a running image):
```python
# runpod-serverless/handler.py
_client = TestClient(app, follow_redirects=False)
```

Commit: `5c646966` on `claude/romantic-cartwright-e37191`.

**Symptoms observed before finding this:**
- OAuth Connect button → browser shows `{"detail":"Not Found"}` JSON at `api-studio.nyptidindustries.com/api/oauth/google/youtube/browser-start`
- Direct curl POST to same URL returned `200 + Studio SPA HTML` (not a 302)
- GET to same URL correctly returned 405 Allow:POST (proving route registered)
- Raw RunPod /run bypassing CF Worker ALSO returned 200 HTML — confirming bug is in backend handler chain, not the Worker

**Why this affects every redirect handler**: see `youtube.py:821` (`_youtube_start_oauth_browser_redirect`), `backend.py:19542` (`_paypal_return`), `billing.py` manage-portal, etc. All return `RedirectResponse(..., status_code=302)`. All broken the same way.

  -F 'metadata={"main_module":"worker-proxy.js","compatibility_date":"2026-04-01","bindings":[]};type=application/json' \

Using filename override silently uploads the OLD code while reporting `"success":true`. ## 8. Gotchas discovered this session

1. **`curl -F "name=@path;filename=override"`** silently uploads OLD file while returning success. Use `-F "basename=@path"` instead.
2. **CF MCP tools are read-only for Workers.** `workers_list`, `workers_get_worker`, `workers_get_worker_code` exist — no `workers_put` or `workers_deploy`. Use REST API with a write-scoped token.
3. **Python not installed** on Casey's Windows machine. `perl` is, and has `JSON::PP` standard. Use perl for JSON manipulation in shell scripts.
4. **`/tmp` paths** differ between Claude's Write tool and bash's `/tmp`. Use `cat <<EOF` heredoc inside a Bash command to create scripts that bash can then read.
5. **Gmail Sent** doesn't have the YouTube API quota form submission. The form uses Google's support flow; only Google's email auto-reply has the quoted-back form contents.
6. **RunPod `/runsync` 30s timeout** hides IN_QUEUE responses behind raw JSON blobs. CF Worker now polls `/status/{id}` for up to 60s to resolve properly.
7. **RunPod account-wide worker quota is 5.** With 4 jrvs-* endpoints (each Max=1), studio-api-ada24 was starved even though it showed `workersMax=2`. Fix was to pause 3 jrvs-* endpoints.
8. **`idleTimeout=5s`** is too aggressive for any real workload. 60s is the sweet spot for OAuth round-trips (which take ~30-60s total from initial click to final callback).
9. **GHCR package visibility** is managed at the package-owner level, not the repo level. Workflow `make-public` step requires a PAT with specific scope, often fails silently. Manual UI toggle is the reliable fix.
10. **Vercel MCP connector** is read-only for deployments — no `deploy_to_vercel` write tool in its exposed surface. Deployment is git-triggered.
11. **TestClient's `follow_redirects=True`** quietly breaks every in-process `RedirectResponse` flow. Root cause of this session's final 3+ hours of debugging.
